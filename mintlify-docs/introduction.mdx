<Note>
**Hammer is a fork of GSD-2.** Same auto-mode loop, different posture: no phase-boundary confirmation prompts, IAM as the only structural guardrail, recover-and-resume after every interruption, and Omega-driven phase synthesis. Internal-implementation surface (CLI binary `gsd`, npm package `gsd-pi`, filesystem paths `.gsd/`, MCP tools `gsd_*`, env vars `GSD_*`) keeps its legacy names — only the product surface (slash commands, chat handle, docs) is rebranded.
</Note>

Hammer is an autonomous coding agent. Describe what you want built, run `/hammer auto`, and walk away. Come back to working software with clean git history.

## What Hammer does

<CardGroup cols={2}>
  <Card title="Autonomous execution" icon="robot">
    A state machine reads your project state, dispatches work to an LLM in fresh context windows, and advances through research, planning, execution, and verification — without phase-boundary confirmation prompts.
  </Card>
  <Card title="Clean git history" icon="code-branch">
    Every task produces a conventional commit. Milestones are squash-merged to main. Your `git log` reads like a changelog.
  </Card>
  <Card title="Cost control" icon="gauge">

  </Card>
  <Card title="Recover-and-resume" icon="rotate">
    Sessions recover from crashes via a recovery Omega manifest. Provider errors auto-retry. Headless mode auto-restarts with exponential backoff. Designed for overnight unattended execution.
  </Card>
</CardGroup>

## How it works

Hammer organizes work into a hierarchy:

```
Milestone  →  a shippable version (4-10 slices)
  Slice    →  one demoable vertical capability (1-7 tasks)
    Task   →  one context-window-sized unit of work
```

The iron rule: **a task must fit in one context window.** If it can't, it's two tasks.

Auto mode loops through this hierarchy:

```
Plan → Execute (per task) → Complete → Reassess Roadmap → Next Slice
                                                            ↓ (all slices done)
                                                    Validate → Complete Milestone
```

Every phase gets a fresh context window with pre-loaded context — no accumulated garbage, no degraded quality. Every phase boundary is synthesized through an **Omega manifest**, gated by the **IAM** policy, and persisted so **recover-and-resume** can reconstruct in-flight work after any interruption.

## What's different from GSD-2

| Surface | GSD-2 | Hammer |
|---------|-------|--------|
| Phase boundaries | Confirmation prompts | Automatic, Omega-driven |
| Subagent dispatch | Permissive | IAM fail-closed gate |
| Crash recovery | Best-effort replay | Recover-and-resume with `RECOVERY_VERDICT` |
| Question count | Heuristic | Per-round Omega synthesis |
| Slash commands | `/gsd …` | `/hammer …` (with `/gsd …` legacy alias) |
| Chat handle (VS Code) | `@gsd` | `@hammer` |
| MCP tool names | `gsd_*` only | `hammer_*` canonical + `gsd_*` alias |
| `.gsd/` directory | Yes | Yes (preserved for compatibility) |
| `gsd-pi` npm package | Yes | Yes (preserved for compatibility) |

## Two ways to work

<Tabs>
  <Tab title="Step mode">
    Type `/hammer` (or `/gsd`) inside a session. Hammer executes one unit at a time, pausing between each so you can review.

    ```bash
    gsd
    /hammer
    ```
  </Tab>
  <Tab title="Auto mode">
    Type `/hammer auto` and walk away. Hammer autonomously researches, plans, executes, verifies, and commits until the milestone is complete — no phase-boundary prompts.

    ```bash
    gsd
    /hammer auto
    ```
  </Tab>
</Tabs>

The recommended workflow: auto mode in one terminal, steering from another.

**Terminal 1 — let it build:**

```bash
gsd
/hammer auto
```

**Terminal 2 — steer while it works:**

```bash
gsd
/hammer discuss    # talk through architecture decisions
/hammer status     # check progress
/hammer capture    # fire-and-forget thoughts
```
